# Skill: Git Master (提交规范专家)

## 核心目标
统一提交质量，保证提交历史可读、可回滚、可审查。

## 提交原则

### 1) 原子化提交（Atomic Commits）
- 一个提交只做一件事。
- 禁止将不相关的改动混在同一个提交中（例如：功能 + 重构 + 格式化）。
- 若改动跨度较大，先拆分为多个可独立理解、独立回滚的提交。

### 2) 约定式提交（Conventional Commits）
- 统一格式：`<type>(<scope>): <subject>`
- 常用 type：
  - `feat`: 新功能
  - `fix`: 缺陷修复
  - `refactor`: 重构（不改变外部行为）
  - `docs`: 文档变更
  - `test`: 测试相关
  - `chore`: 杂项维护（依赖、脚本、构建）

### 3) 中文 message 约束（你新增的规则）
- 冒号 `:` 后面的 `<subject>` 必须使用中文。
- 示例：
  - `feat(auth): 新增短信登录能力`
  - `fix(api): 修复用户列表分页参数错误`

## 生成提交信息时的执行规则
- 先判断本次改动是否原子化；若不是，先给出拆分建议，再分别生成提交信息。
- 每条提交信息保持简洁，聚焦“为什么做这次改动”。
- 避免笼统描述（如“update code”）。

## 自定义指令

### /prepare-commit
触发后执行以下动作：
1. 分析当前已暂存与未暂存改动。
2. 判断是否满足原子化提交要求。
3. 若不满足，给出建议拆分方案（按提交粒度分组）。
4. 为每组改动生成一条符合规范的提交信息（`type(scope): 中文描述`）。

### /conventional-commit
触发后执行以下动作：
1. 基于当前改动内容，直接生成 1-3 条候选提交信息。
2. 每条都必须遵循 `type(scope): 中文描述`。
3. 若 scope 不明确，可省略 scope，格式为 `type: 中文描述`。
4. 若检测到改动不原子化，先提示拆分，再分别生成提交信息。

### /commit-check
触发后执行以下动作：
1. 检查你提供的提交信息是否符合 Conventional Commits。
2. 检查 `:` 后是否为中文描述。
3. 检查当前改动是否可能违反原子化提交原则。
4. 返回“通过/不通过”与修正建议。
