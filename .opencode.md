# OpenCode Project Instructions

本文件用于定义本仓库在 OpenCode 中的项目级行为约束。

## 全局输出规则
- 每次任务响应末尾都必须包含 `## 本次执行总结` 小节。
- 小节至少包含：
  - 已完成事项
  - 关键变更
  - 验证结果

## Git 提交规则
- 提交必须遵循 Conventional Commits。
- 提交应保持原子化：一个提交只做一件事。
- 提交信息格式：`<type>(<scope>): <subject>` 或 `type: <subject>`。
- `:` 后面的 `<subject>` 必须为中文。

## AI 代码注释规则
- AI 生成或修改代码时，必须补充“合适且必要”的注释，目标是让后续维护者和 AI 能快速理解。
- 注释应优先解释“为什么这么做”和关键约束，不重复代码字面含义。
- 以下场景必须加注释：
  - 复杂业务分支、边界条件、容错与回退逻辑。
  - 与外部系统交互的关键约束（协议、字段映射、重试/幂等规则）。
  - 性能或安全相关的非常规实现（缓存、锁、鉴权、输入校验等）。
  - 临时兼容方案与技术债（需包含 TODO 与后续处理方向）。
- 以下场景不应加冗余注释：
  - 简单 CRUD、直白变量赋值、与函数名重复的解释。
  - 明显的一行语义，不增加额外信息的注释。
- 注释风格要求：
  - 使用简体中文，简洁明确，避免长段落。
  - 优先使用函数/代码块上方注释；仅在必要时使用行尾注释。
  - TODO 格式统一为：`TODO(负责人或模块): 具体动作`。
- 每次代码变更完成后，在回复的“本次执行总结”中单独说明：
  - 哪些文件新增了注释。
  - 为什么这些位置需要注释。

## Slash Commands
当用户输入以下命令时，按对应流程执行：

### /prepare-commit
1. 查看当前 staged/unstaged 变更。
2. 判断是否满足原子化提交。
3. 若不满足，先给拆分方案。
4. 为每个拆分生成 `type(scope): 中文描述` 的提交信息。

### /conventional-commit
1. 基于当前变更直接给出 1-3 条候选提交信息。
2. 格式必须为 `type(scope): 中文描述` 或 `type: 中文描述`。
3. 若变更不原子化，先提示拆分，再分别给提交信息。

### /commit-check
1. 检查给定提交信息是否符合 Conventional Commits。
2. 检查 `:` 后是否中文。
3. 结合当前改动判断是否违反原子化原则。
4. 返回“通过/不通过”和修正建议。
