# 提示词模板库

说明：每条模板必须包含四段：输入、输出、禁区、验收。

## Explore（调研）

### E1 模块职责盘点
- 输入：扫描目标仓库，识别模块、入口、依赖与边界。
- 输出：模块职责表（模块/职责/关键文件/未知项）。
- 禁区：无代码证据不得下结论。
- 验收：顶层模块覆盖完整。

### E2 API 清单提取
- 输入：提取 method、path、request、response、error code。
- 输出：按域分组的 API 矩阵。
- 禁区：不得凭经验补接口。
- 验收：user/canvas/share/ai/health 全覆盖。

### E3 WS 协议发现
- 输入：定位 WS type、payload、服务端处理、前端消费点。
- 输出：按 type 的契约表与示例消息。
- 禁区：不得虚构消息类型。
- 验收：ping/add/update/delete/flush 全量存在。

### E4 SSE 协议发现
- 输入：定位 endpoint、event、close/error 行为。
- 输出：EventSource 可消费规范草案。
- 禁区：不得改名既有事件语义。
- 验收：push/close/error 定义清晰。

### E5 配置与环境变量盘点
- 输入：收集 env 变量及消费位置。
- 输出：变量表（默认值/必填/敏感级别）。
- 禁区：不得暴露真实密钥。
- 验收：app/db/redis/session/minio/ai 分组齐全。

### E6 数据库结构盘点
- 输入：提取表、字段、索引、外键、默认值。
- 输出：as-is 数据库文档与迁移风险点。
- 禁区：不得脱离 schema 或 migration 猜测。
- 验收：核心表与约束完整。

### E7 依赖与调用链盘点
- 输入：分析模块 import/use 关系。
- 输出：依赖图与高风险耦合点。
- 禁区：不得写无证据跨层关系。
- 验收：高风险耦合有定位和说明。

### E8 兼容差异报告
- 输入：对比旧行为与目标行为。
- 输出：必须保持/可调整/禁止变更列表。
- 禁区：不得降低兼容而不说明原因。
- 验收：按前端破坏风险分级。

## Spec（规格）

### S1 OpenSpec 基线
- 输入：调研结论。
- 输出：范围、契约、约束、验收标准。
- 禁区：不得出现不可验证描述。
- 验收：每段都可判定通过/阻断。

### S2 ADR 决策文档
- 输入：候选技术栈与约束。
- 输出：候选对比、取舍、最终决策、替代与回滚。
- 禁区：不得只给结论不写 why-not。
- 验收：决策可追溯。

### S3 API 兼容红线
- 输入：前端依赖行为。
- 输出：必须/可选/禁止清单。
- 禁区：不得在必须项中改 path/method。
- 验收：envelope、401、cookie 规则明确。

### S4 权限矩阵
- 输入：资源操作与 owner/edit/view 角色。
- 输出：接口级放行/拒绝矩阵。
- 禁区：不得同接口混用冲突规则。
- 验收：可直接用于 guard/policy 编码。

### S5 错误码映射
- 输入：旧错误码与前端动作。
- 输出：旧码到新异常到 HTTP 到前端动作映射。
- 禁区：不得一号多义。
- 验收：高频错误码语义稳定。

### S6 WS 契约规格
- 输入：现有 WS 协议与目标一致性规则。
- 输出：消息结构、冲突处理、flush 策略。
- 禁区：不得允许静默覆盖冲突。
- 验收：每个 type 有样例。

### S7 SSE 契约规格
- 输入：前端 EventSource 解析逻辑。
- 输出：事件时序、异常处理、重连策略。
- 禁区：不得使用非标准 SSE 帧。
- 验收：浏览器原生 EventSource 可直接消费。

### S8 任务图与回滚点
- 输入：全量范围和依赖关系。
- 输出：原子步骤图、Gate、回滚点。
- 禁区：不得出现跨步混合任务。
- 验收：每步独立可验收。

## Code（实现）

### C1 原子实现
- 输入：当前步骤范围与验收项。
- 输出：最小差异实现。
- 禁区：不得加入后续步骤功能。
- 验收：构建通过且行为匹配。

### C2 模块骨架
- 输入：目标模块与分层规则。
- 输出：controller/service/dto/repository 骨架。
- 禁区：骨架步骤不得写业务逻辑。
- 验收：编译通过。

### C3 接口实现
- 输入：路由契约与 DTO 校验要求。
- 输出：统一 envelope + 错误映射接口。
- 禁区：不得改变响应结构语义。
- 验收：请求/响应/状态码符合契约。

### C4 鉴权与装饰器
- 输入：session 鉴权与白名单规则。
- 输出：AuthGuard、Public、CurrentUser。
- 禁区：不得硬编码绕过。
- 验收：受保护路由正确拦截。

### C5 仓储分层
- 输入：现有 service 与数据访问逻辑。
- 输出：仓储层抽离，行为不变。
- 禁区：不得在抽离时改变语义。
- 验收：原有测试与联调不回归。

### C6 迁移生成
- 输入：schema 变更集。
- 输出：migration 文件与执行步骤。
- 禁区：无计划不得做破坏性变更。
- 验收：迁移命令可执行。

### C7 韧性与降级
- 输入：超时、重试、兜底策略。
- 输出：有边界的重试与降级分支。
- 禁区：不得无限重试。
- 验收：失败路径可预测且可观测。

### C8 配置默认值
- 输入：环境变量矩阵与本地启动要求。
- 输出：类型化配置与最小默认值。
- 禁区：不得写入真实密钥。
- 验收：最小 env 可本地启动。

## Test（测试）

### T1 先红后绿
- 输入：当前步骤验收标准。
- 输出：先失败测试，再最小修复。
- 禁区：不得写无断言“假绿”测试。
- 验收：修复前失败，修复后通过。

### T2 权限单测
- 输入：owner/edit/view + 未登录场景。
- 输出：放行/拒绝分支覆盖。
- 禁区：不得只测 happy path。
- 验收：关键分支覆盖完整。

### T3 Session e2e
- 输入：register/login/check-auth/logout 流程。
- 输出：携带 cookie 的 e2e 测试。
- 禁区：不得共享脏会话状态。
- 验收：401 与成功路径都可复现。

### T4 WS 契约脚本
- 输入：双客户端同房间场景。
- 输出：广播与冲突处理断言脚本。
- 禁区：不得只靠固定 sleep 同步。
- 验收：重复运行稳定通过。

### T5 SSE 契约脚本
- 输入：流接口与事件顺序要求。
- 输出：push/close/error 序列断言。
- 禁区：不得忽略格式错误事件。
- 验收：事件顺序和数据合法。

### T6 上传校验 e2e
- 输入：类型、大小、权限规则。
- 输出：成功与失败用例集。
- 禁区：不得依赖临时本地路径。
- 验收：非法上传按预期拒绝。

### T7 回归入口脚本
- 输入：lint/unit/e2e/ws/sse 流程。
- 输出：一键执行且失败即停。
- 禁区：不得吞掉错误退出码。
- 验收：任一失败返回非 0。

### T8 测试缺口报告
- 输入：环境阻塞或依赖缺失。
- 输出：缺口、替代验证、后续动作。
- 禁区：不得把无法执行标记为通过。
- 验收：阻塞项清晰可追踪。

## Review（审查）

### R1 当前步审查
- 输入：当前步骤 diff 与验收标准。
- 输出：按严重级排序的问题清单。
- 禁区：不得以风格问题淹没功能风险。
- 验收：优先暴露 bug/回归/缺测。

### R2 兼容性审查
- 输入：API/WS/SSE 变更与红线规则。
- 输出：接口级破坏风险报告。
- 禁区：不得忽略潜在破坏性变更。
- 验收：必须兼容项保持稳定。

### R3 安全审查
- 输入：鉴权、校验、日志、密钥处理。
- 输出：漏洞清单与修复建议。
- 禁区：不得输出敏感明文。
- 验收：无未处置高危风险。

### R4 性能审查
- 输入：DB、WS、SSE 热路径。
- 输出：热点与容量风险分析。
- 禁区：不得放过无界增长逻辑。
- 验收：热点均有缓解建议。

### R5 错误码一致性审查
- 输入：异常实现与错误码映射表。
- 输出：冲突与漂移检查结果。
- 禁区：不得复用同码表达不同语义。
- 验收：冲突项为 0。

### R6 回滚可行性审查
- 输入：回滚模板与当前变更。
- 输出：可执行回滚清单。
- 禁区：不得写不可验证回滚步骤。
- 验收：回滚步骤可落地执行。

### R7 证据完整性审查
- 输入：步骤日志、命令输出、差异证据。
- 输出：通过或阻断结论。
- 禁区：不得缺证据放行。
- 验收：证据字段齐全。

### R8 Gate 结论
- 输入：上述审查结论。
- 输出：Pass/Block/Partial 与原因。
- 禁区：不得给模糊结论。
- 验收：下一步动作明确。

## Retro（复盘）

### RT1 结果复盘
- 输入：目标与实际结果。
- 输出：有效动作与失败点。
- 禁区：不得空泛总结。
- 验收：至少 1 条可执行改进项。

### RT2 提示词 A/B
- 输入：优化前后提示词。
- 输出：差异与质量变化。
- 禁区：不得无证据宣称改进。
- 验收：至少 1 条规则升级。

### RT3 浪费分析
- 输入：无效尝试与返工记录。
- 输出：避免清单与预防规则。
- 禁区：不得只归因不改进。
- 验收：有可执行预防动作。

### RT4 上下文缺口分析
- 输入：因信息缺失导致阻塞的点。
- 输出：下一阶段上下文清单。
- 禁区：不得重复遗漏同类信息。
- 验收：清单可复用。

### RT5 波动与不稳定分析
- 输入：不稳定测试或流程波动。
- 输出：稳定化措施。
- 禁区：不得以重复重跑代替修复。
- 验收：不稳定根因被识别。

### RT6 可复用资产沉淀
- 输入：当前步骤产物。
- 输出：可复用模板/脚本/检查单。
- 禁区：不得沉淀无复用价值资产。
- 验收：至少 1 个可复用资产。

### RT7 反模式记录
- 输入：执行中出现的坏模式。
- 输出：反模式与替代方案。
- 禁区：不得只写问题不写替代。
- 验收：替代方案可验证。

### RT8 下一步准备度
- 输入：当前 Gate 结果与阻塞项。
- 输出：可启动条件与前置事项。
- 禁区：不得在阻塞未清时启动下一步。
- 验收：下一步启动条件明确。

## Retro-1 提示词优化补丁（2026-04-02）

### OP1 依赖引入增强（优化 C1/C2）
- 优化前：「安装 xxx 依赖」
- 优化后：「安装 xxx 依赖，验证导入方式（default vs named export），立即执行 npm run build 确认 import 兼容」
- 适用步骤：所有引入新 npm 包的 Code 步骤

### OP2 枚举/常量全量提取（优化 E2/E5）
- 优化前：「识别关键错误码示例」
- 优化后：「提取 ResponseCode/ErrorCode 完整源文件，输出全量编号-语义-前端消费点映射表」
- 适用步骤：Phase A 所有枚举/常量盘点步骤

### OP3 e2e 测试环境前置（优化 T3）
- 优化前：「编写 e2e 测试」
- 优化后：「编写 e2e 测试。前置要求：1) 配置 test/setup-env.ts 设置所有 required 环境变量；2) 使用 overrideProvider(PrismaService).useValue(mock) 隔离数据库；3) 手动注册 session 中间件、ValidationPipe、全局拦截器/过滤器」
- 适用步骤：所有 NestJS e2e 测试编写步骤

### OP4 session fixation 防护（优化 C4）
- 优化前：「登录后写入 session」
- 优化后：「登录/注册成功后先调用 req.session.regenerate() 重新生成 session ID，再写入 userId/username」
- 适用步骤：所有涉及 session 写入的认证步骤
