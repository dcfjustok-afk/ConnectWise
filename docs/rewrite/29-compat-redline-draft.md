# Step 029 兼容红线初稿

目标：定义迁移期间不可破坏的兼容红线，作为 Phase B-D 的硬门禁。

## 一、红线定义（不可破坏）

| 红线编号 | 红线内容 | 触发即阻断 |
|---|---|---|
| RL-01 | 既有登录态能力不退化（`/user/login` `/user/check-auth` `/user/logout`） | 是 |
| RL-02 | API 路径兼容（含 `/api` 前缀策略一致） | 是 |
| RL-03 | 画布核心 CRUD 行为与返回码不倒退 | 是 |
| RL-04 | Share 权限语义保持 owner/edit/view 一致 | 是 |
| RL-05 | WS 消息类型与冲突回包语义兼容 | 是 |
| RL-06 | SSE 事件流行为兼容（至少 `push/close/error` 语义不变） | 是 |
| RL-07 | 上传接口兼容 multipart，失败语义可识别 | 是 |
| RL-08 | 会话鉴权基线不弱化（Session + Redis） | 是 |

## 二、红线判定标准

1. 前端关键路径联调脚本全部通过。
2. 既有错误码映射不中断，客户端可继续识别。
3. 协议层（WS/SSE）客户端不需要重写消费模型即可工作。

## 三、阻断与回滚规则

- 任一红线失败：当前步骤直接 Block，不得推进下一步骤。
- 修复顺序：先恢复兼容，再做优化。
- 回滚优先级：API > 协议 > 权限 > 性能。

## 四、后续动作

- Step 030-036 的技术栈问答必须服务于红线达成，不允许引入与红线冲突的方案。
